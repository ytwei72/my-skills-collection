# Markdown → 智能体响应结果形式的 HTML（摘要卡片 + 详情双页）

## 何时使用

- 用户**明确**提到「智能体响应结果形式的 html」「智能体响应 HTML」「摘要卡片 + 详情 HTML」「双页 HTML（卡片入口 + 完整版）」。
- 用户想要一个**像聊天里 AI 卡片回复**那样的高度概括入口，再保留一份完整、可阅读的富页面给愿意深入查看的人。
- 用户已经提供了 `.md` / `.markdown` 文件路径或内容，希望以这种形式呈现。

> 与 `md-to-rich-html` 的差异：本技能 **只接受 Markdown**，并且**一次产出两个 HTML** 互相链接。如果用户只要单文件富 HTML、或源不是 Markdown，请改用 `md-to-rich-html`。

## 输入约束

- **仅支持** `.md` / `.markdown`。其他扩展名（`.txt` / `.html` / `.csv` / `.json` / `.docx` …）一律**拒绝**，提示用户先转成 Markdown 或改用 `md-to-rich-html`。
- 单次任务**一个 Markdown 源**对应**一组双页 HTML**；若用户给多个 md 文件，则**逐文件**重复执行。

## 产出物

每个源 Markdown 产出 **两个** 同目录、同前缀的 HTML 文件（默认与源文件同目录）：

| 文件 | 角色 | 命名约定 |
|------|------|---------|
| `<name>.summary.html` | **入口卡片**：概要 + 关键信息 + 重要结论 + 关键数据 | `<源文件名>.summary.html` |
| `<name>.detail.html`  | **完整富页**：等同 `md-to-rich-html` 风格的整篇渲染 | `<源文件名>.detail.html` |

- 若用户指定了输出路径/前缀，沿用用户指定值；两份文件**必须放在同一目录**，使用**相对路径**互相链接，保证拷走文件夹后链接仍然可用。
- 严禁使用 `file://` 绝对路径或任何外网 URL 做跳转。

## 互相跳转规范

### 摘要 → 详情

- `summary.html` **整张卡片**（或卡片内显著的「查看完整版」按钮）使用 `<a href="<name>.detail.html">…</a>` 跳转。
- 链接需保留键盘可达性（`a` 元素本身可聚焦）、有 `aria-label`，且不要嵌套 `a` in `a`。
- 鼠标悬停时给一个轻微 hover 反馈（`transform: translateY(-2px)` 或阴影加深），暗示可点击。

### 详情 → 摘要

- `detail.html` **左上角固定**一个返回按钮：
  - 推荐写法：`<a class="back-btn" href="<name>.summary.html" aria-label="返回摘要卡片">← 返回</a>`。
  - CSS：`position: fixed; top: clamp(12px,2vw,20px); left: clamp(12px,2vw,20px); z-index: 50; padding: 8px 14px; border-radius: 999px; background: var(--surface); color: var(--text); border: 1px solid var(--border); box-shadow: var(--shadow); text-decoration: none;`
  - 深浅色模式都要可读；hover/focus 给反馈；窄屏下尺寸自动收敛但**不要遮挡正文标题**（必要时给 `body` 加 `padding-top`）。

## 摘要卡片（summary.html）写作要求

> 这是**面向决策者/扫读者**的一页。**默认只占一屏左右**，主打「极简、清爽、扫一眼就懂」，不要把整篇 MD 抄进来。

### 三区域结构（必须遵守）

整张卡片**从上到下**分成三个**视觉上明显区分**的区域，三区之间用**底色 / 分隔线 / 留白**形成清晰边界：

```
┌─────────────────────────────────────┐
│ ① 标题区（Title）                  │  紧凑、纯净、视觉权重最高的一行
├─────────────────────────────────────┤
│ ② 副标题区（Subtitle / 概要说明） │  一句话摘要，柔和强调色背景
├─────────────────────────────────────┤
│ ③ 内容区（Content）               │  关键结论 + 可选指标 + CTA
└─────────────────────────────────────┘
```

#### ① 标题区

- 内容：**左侧主标题 + 右侧 CTA**，构成一个两端对齐的「标题栏」。
  - **左**：主标题（来自 MD 的 `h1`；无 `h1` 则取首个 `h2` 或文件名）。
  - **右**：CTA 按钮，**文案固定为「查看全文 →」**（不要写"查看完整说明 / 查看详情 / 阅读全文"等其他变体）；语义上仍由整卡 `<a class="card-link">` 承载跳转，标题栏右侧的 CTA 是一个**视觉元素**（用 `<span class="cta">` 或装饰性 `div`），**不要**在 `<a>` 内嵌套另一个 `<a>`。
- **禁止**在标题区写：「摘要卡」「Summary」之类的 pill 标签、来源、文件路径、作者、更新时间等元信息——标题区**只放标题和右侧 CTA**。
- 视觉：背景 `var(--surface)`；标题字号 `clamp(1.5rem, 4.6vw, 2rem)`，字重 800；CTA 字号 `0.88~0.92rem`，圆角 999px，颜色用强调色或强调色边框；标题与 CTA 之间 `gap: clamp(12px, 3vw, 20px)`；窄屏（≤ 480px）下 CTA 可缩成只有"→"或换行到标题下方，但**不要**消失。
- 比例：约占整卡 22%~28%。

#### ② 副标题区（概要说明）

- 内容：**1 句话**（最多 2 行）回答「这篇文档讲什么 / 解决什么问题」；句末**不强行加句号**也可，但避免半截话。
- 视觉：背景**轻强调色** `var(--accent-soft)` 或 `var(--surface-2)`；可在左侧加一条 3~4px 强调色竖条作为「引子」；正文色用 `var(--text)`，副信息用 `var(--muted)`；**不要堆指标也不要列要点**——这里只放一句概要。
- 比例：在三区中**最薄**，约占整卡 14%~20%。

#### ③ 内容区

- 内容（按需，最多保留以下 2 类，**能少则少**）：
  1. **关键结论**：3~5 条 bullet/pill；每条 ≤ 20 字；优先用「动作句 / 断言句」；可用模式色或主题色 pill 做前置标签。
  2. **关键指标**（可选）：MD 里真有数字才放；**≤ 3 个**指标卡，宁可不放也不编造。
- **不再**在内容区放 CTA——CTA 已经在标题区右侧承担；这里**只承载内容**，避免重复引导。
- 视觉：背景 `var(--surface-2)` 或与标题区相同的 `var(--surface)`（与副标题区区分即可）；项目之间用 8~10px gap；最后一项与卡片底部留 18~24px 呼吸。
- 比例：在三区中**最大**，约占整卡 56%~62%。

### 内容精简原则

- **能砍就砍**：每条结论 ≤ 20 字；副标题 ≤ 35 字；指标卡 ≤ 3 个；不要把详情页里的所有亮点都塞进来。
- **不重复说**：副标题已经说过的话，不要在结论里再说一遍。
- **去除元信息**：标题区不写"摘要卡 / 来源 / 文件名 / 更新时间 / 作者 / 版本号"。这些都属于详情页或页脚，不在摘要卡顶部出现。
- **CTA 只放一处**：固定在标题区右侧，**文案固定「查看全文 →」**；内容区与副标题区**禁止**再出现「查看完整说明 / 阅读全文 / 点击查看 …」之类的二次 CTA。
- **指标取舍**：MD 中没有可信数字，**直接不放指标卡**；不要用"覆盖 X 种""更新 YYYY-MM-DD"等无信息量的"凑数指标"占位。
- **MD 信息量极少时**：摘要卡可只保留三区简化版——「标题（带右侧"查看全文"CTA）+ 一句话副标题 + 空内容区或一两条断言句」。

### 视觉要求（整卡）

- **整卡可点击**：`<a class="card-link">` 包裹整张 `<article class="card">`；hover 时整卡轻微上浮（`translateY(-3px)`）+ 阴影加深 + CTA 变实色，作为可点击线索。
- **尺寸**：`max-width: 640~720px`，居中；整卡圆角 18~20px；柔和阴影 `var(--shadow)`。
- **三区分隔**：优先**用底色差异**区分三区；其次可加 `border-top: 1px solid var(--border)`；不要用粗分隔线。
- **配色清爽**：与 detail 页**共用同一套 CSS 变量**；卡顶可加一条 3~4px 强调色（或多色 gradient）作为点缀，但**只在卡片顶部一条**，不要四处铺色。
- **无障碍**：`<a>` 上写 `aria-label`；标题用 `id` 关联 `aria-labelledby`；颜色对比度达标。
- **不要**：复杂 SVG、目录、完整段落、代码块、面包屑导航；这些都属于详情页。

### 抽取规则（从 MD → 摘要内容）

- **副标题**：优先用 MD 开头的引述/导语；其次用 `## 总结 / 结论 / 摘要 / TL;DR` 之下的首句；都没有则用一句话概括 `h1` + 主要 `h2` 的主题。
- **关键结论**：优先抓 MD 中带 `**强调**`、`> 引用块`、`!important`、`✅` 等标记的句子；或每个一级章节首段的断言。
- **数据指标**：在表格、`**数字+单位**`、`数字% / 数字倍 / N 次 / N 小时` 等模式里抓；**MD 没有数字 → 不要放指标卡**。

## 详情页（detail.html）写作要求

- 整体规则**完全沿用** `md-to-rich-html`（语义化标签、响应式、深浅色变量、内联 SVG 图表等），见仓库里 `.cursor/skills/md-to-rich-html/SKILL.md` 与 `reference.md`。
- 强制差异点：
  1. 左上角必须有「返回」按钮，且不能被 `header.hero` 遮挡（必要时给 hero 加 `padding-top` 或让按钮 `z-index` 高于 hero）。
  2. 顶部建议加一个细的"你正在查看完整版"提示条或面包屑（可选）。
  3. 文件名后缀必须是 `.detail.html`，并保证 summary 页相对路径能跳转过来。

## 工作流程

1. **校验输入**：确认是 `.md` / `.markdown`；多文件则记录列表，逐个处理；非 md 直接拒绝并给出迁移建议。
2. **解析归一化**：得到 MD 的标题树 + 块级元素 + 强调；同时抽取摘要素材（结论 / 数据 / 关键句）。
3. **生成 detail.html**：按 `md-to-rich-html` 风格完整渲染整篇；左上角写入"返回"按钮，`href` 指向 `<name>.summary.html`。
4. **生成 summary.html**：按上节"摘要卡片写作要求"装配；整卡 `<a>` 指向 `<name>.detail.html`。
5. **统一主题变量**：两页共用同一套 CSS 变量名（建议直接复制变量块），保证视觉一致与可换肤。
6. **本地自检**：
   - 两个文件**同目录**；
   - 相对链接拼写一致（大小写、扩展名）；
   - 在 detail.html 顶部模拟"返回"是否被遮挡；
   - 在 summary.html 检查"指标卡"数字是否真的来自 MD，避免幻觉；
   - 在 summary.html 检查**三区域**是否在底色/留白上明显可区分；标题区**没有**"摘要卡 / 来源/文件名/更新时间"等元信息；标题区右侧**有**「查看全文 →」CTA 且全卡只此一处；内容总量是否做到了精简。
7. **交付说明**：告知用户两个文件路径，提示"先打开 summary.html"。

## 命名与路径示例

源文件：`docs/AI工具应用情况工作汇报.md`

产出：

- `docs/AI工具应用情况工作汇报.summary.html`（用户先打开这个）
- `docs/AI工具应用情况工作汇报.detail.html`

跳转：

- `summary.html` 内：`<a class="card-link" href="AI工具应用情况工作汇报.detail.html">…</a>`
- `detail.html` 内：`<a class="back-btn" href="AI工具应用情况工作汇报.summary.html">← 返回</a>`

> 注意中文文件名在跳转链接里使用相对路径即可，浏览器会自动 URL 编码；不要手动写 `%E2%80%A6` 等编码，保留原文件名最直观。

## 反幻觉与可靠性

- **绝不**在摘要里写 MD 中没有的数字、结论、人名、日期。
- **绝不**虚构"提效 60%"这类指标——只有 MD 里写了才能放，否则把指标卡换成关键词 pill。
- 如果 MD 内容太少不足以构成摘要卡，**宁愿摘要卡只有标题 + 一句话 + CTA**，也不要灌水。
- 如果 MD 里包含敏感数据（手机号、邮箱、密钥），摘要卡可打码；详情页按用户要求处理。

## 拒绝场景

- 用户给 `.docx` / `.txt` / 网页链接 / 图片 / 截图作为源：**拒绝**，提示「本技能仅支持 Markdown 源文档，请先转为 .md 或改用 md-to-rich-html」。
- 用户要求合并多个 md 进同一对页面：**询问确认**，并以「合并后的虚拟 md」作为输入再走流程；不要静默合并。
- 用户要求把摘要与详情合进一个 html：建议改用 `md-to-rich-html`，本技能产出**始终是两个 html**。

## 与其他技能的关系

- `md-to-rich-html`：单文件富 HTML、支持多源格式。本技能是它的「卡片入口 + 详情」双页变种。
- `feishu-doc-writer` / `feishu-doc-reader`：若用户要把生成的 HTML 同步到飞书文档，先生成本地双页 HTML，再视需要把内容（不是 HTML 本身）整理为 MD 推送到飞书；不要把整段 HTML 直接灌进飞书 docx。

## 交付话术

- 简洁告知两个文件的路径；
- 提示「双击 `*.summary.html` 浏览器打开，点击卡片可进入完整版，完整版左上角可返回」；
- 不要重复贴大段 HTML 源码给用户，除非用户明确要求查看代码。
